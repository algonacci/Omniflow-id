# SEO Hardening Plan — Omniflow.id

**Last updated:** 2026-04-19
**Audience:** AI workers picking this up cold. Read top-to-bottom before touching code.

---

## 0. Project context (read first)

Omniflow.id is a **multilingual (EN/ID/ZH) React + Vite SPA** selling an Indonesian ERP SaaS with 25 modules. Primary market = Indonesia. Stack details:

- Routes live in `src/App.tsx`, duplicated across `/en/*`, `/id/*`, `/zh/*`, plus legacy `/*`.
- i18n strings in `src/i18n/locales/{en,id,zh}.json`.
- 25 modules + 4 categories are defined as single source of truth in `src/lib/modules.ts`.
- SEO is injected via two components:
  - `src/components/SEOHead.tsx` — per-route `<title>`, meta, OG, Twitter, hreflang, canonical, WebPage + Organization JSON-LD.
  - `src/components/StructuredData.tsx` — emits additional JSON-LD (`Organization`, `WebSite`, `SoftwareApplication`, `Service`, `Article`) when dropped in a page.
- Default SEO baseline also sits inline in `index.html`.
- Sitemaps are regenerated by `scripts/generate_sitemaps.cjs` (hooked to `prebuild` in `package.json`). Source of truth for URLs is that script's `MODULE_SLUGS` / `STATIC_PAGES`.
- `public/robots.txt` points to all 4 sitemaps.

**Target market**: Indonesia-first. Primary language for SEO investment = `id`. `en` is secondary (regional/international), `zh` is a nice-to-have.

## 2. Known issues to fix (prioritized)

### P0 — Critical correctness & Google policy risks

#### [P0-1] Remove fake `aggregateRating` (risk: Google manual action)
- **Files:** `src/components/StructuredData.tsx` (lines ~162-168 and ~237-241).
- **Issue:** Hardcoded `"ratingValue": "4.8", "reviewCount": "1250"` with no real reviews on the page. Google's structured-data policy treats fabricated ratings as spam; pages can get demoted or hit with a manual action.
- **Fix:** Delete both `aggregateRating` blocks. Also delete the inline `review` block (lines ~222-236) with "Sarah Johnson". Only re-add when real testimonial data exists on the rendered page and can be sourced from a real system.

#### [P0-2] Fix inconsistent business NAP (Name/Address/Phone)
- **Files:**
  - `index.html` lines ~76-88: "Jl. Sudirman No. 123, Jakarta Pusat 10220", phone `+62-21-1234-5678` (placeholder).
  - `src/components/SEOHead.tsx` lines ~339-357: "Jl. Letjen S. Parman No. 28, Grogol Petamburan 11470", phone `+62 821 2560 9413`.
  - `src/components/StructuredData.tsx` lines ~62-88: "Jl. Sudirman No. 123" again.
- **Issue:** Three different addresses across three files. Inconsistent NAP = Google trust penalty + user confusion. The `+62-21-1234-5678` phone is a placeholder, must go.
- **Fix:** Ask the user for the correct address+phone, then normalize all three files to match. Until confirmed, **block deploy of structured-data changes** — inconsistent NAP is worse than none. Use `+62 821 2560 9413` as a placeholder *only if user confirms*.

#### [P0-3] Canonicalize Organization JSON-LD (currently duplicated 3x)
- **Files:** `index.html` (inline `<script>` ~65-104), `SEOHead.tsx` (`data-seo="organization"` ~319-366), `StructuredData.tsx` (`getOrganizationData` ~46-122).
- **Issue:** Three separate Organization graphs on the same page, partially contradicting each other. Google picks one unpredictably and the others become noise.
- **Fix:** Keep the inline one in `index.html` as the static fallback for crawlers that don't run JS. Remove the duplicate emission from `SEOHead.tsx`. Reserve `StructuredData.tsx type="Organization"` only for the homepage (where it's already mounted). Make sure all three files read from one shared JS constant (e.g., add `src/lib/org.ts` exporting `ORG_PROFILE`).

#### [P0-4] Title/description: add primary keyword to home
- **Files:** `src/i18n/locales/{en,id,zh}.json` → `hero.title`, `hero.subtitle`.
- **Issue:** Current ID `hero.title` doesn't contain "aplikasi ERP" — the #1 keyword we want to rank for.
- **Fix:** Rewrite per § 3.1 below. Keep title ≤ 60 chars, description ≤ 160 chars.

#### [P0-5] Fake `installUrl` / `downloadUrl` in SoftwareApplication
- **File:** `src/components/StructuredData.tsx` lines ~180-182.
- **Issue:** `installUrl: "https://omniflow.id/signup"` and `downloadUrl` point to pages that don't exist (check `src/App.tsx` — no `/signup` or `/documentation` route). Google may flag.
- **Fix:** Either remove these fields, or point them to `https://omniflow.id/{lang}/contact`.

### P1 — High impact

#### [P1-1] Per-module page SEO (currently none customized)
- **Files:** Every `src/pages/modules/*/index.tsx` — inspect whether `<SEOHead>` is receiving module-specific `title`/`description`/`keywords` or defaulting to generic home values.
- **Fix:** For each of the 25 modules, set:
  - `title={t("<module>.seo.title")}` — format: `{keyword} — Omniflow.id` (e.g., `Aplikasi HRIS Terintegrasi — Omniflow.id`).
  - `description={t("<module>.seo.description")}` — include primary keyword + one support phrase + CTA, under 160 chars.
  - `keywords={primary + 2-3 secondary}` — comma-separated, not stuffed.
- **Add i18n keys:** for each module in `{en,id,zh}.json`, add `<moduleKey>.seo.{title,description,keywords}`. See § 3.2 for templates.

#### [P1-2] Add `BreadcrumbList` schema to module pages
- **Why:** Breadcrumbs render in Google SERP snippets and raise CTR.
- **Where:** In each module index page (or better, centralize in `SEOHead.tsx` via a new `breadcrumb` prop that takes an array of `{name, url}`).
- **Shape:**
  ```json
  {
    "@type": "BreadcrumbList",
    "itemListElement": [
      {"@type": "ListItem", "position": 1, "name": "Home", "item": "https://omniflow.id/id"},
      {"@type": "ListItem", "position": 2, "name": "Modules", "item": "https://omniflow.id/id/modules"},
      {"@type": "ListItem", "position": 3, "name": "Aplikasi HRIS", "item": "https://omniflow.id/id/modules/hris"}
    ]
  }
  ```

#### [P1-3] `FAQPage` schema where FAQs exist
- **Where:** `src/components/FAQ.tsx` renders on homepage. Probably also module pages (check `src/pages/modules/*/components/`).
- **Fix:** Emit `FAQPage` JSON-LD alongside the FAQ UI. Only include questions that are **actually visible** on the page (Google strips hidden Q&A).

#### [P1-4] Replace Unsplash OG image with branded asset
- **Files:** `index.html` line 34, `SEOHead.tsx` line 23, `StructuredData.tsx` line 176.
- **Issue:** Generic Unsplash photo → low brand memorability on social shares and SERP thumbnails.
- **Fix:** Create 1200×630 branded OG image (logo + tagline + product screenshot) at `public/og-default.png`. Reference `https://omniflow.id/og-default.png`. Consider per-locale variants.

#### [P1-5] Internal linking across module pages
- **Why:** Flat site structure — every module is equally distant from home. Need internal link equity flow.
- **Fix:** In each module page, add a "Related modules" section (2-3 cards) that links to other modules in the same `moduleCategories` group (from `src/lib/modules.ts`). Implementation hint: filter same-category modules except self, take first 3.

#### [P1-6] Kill Google manual-action risks in robots.txt
- **File:** `public/robots.txt`.
- **Issue:** `Crawl-delay: 1` — most engines ignore this; Bing respects it but it's fine. `Disallow: /_next/` is Next.js-specific and irrelevant here, though harmless.
- **Fix:** Remove `/_next/` line, keep `/admin/`, `/api/`. Add `Disallow: /*?*` if/when query-string URLs start appearing (currently none).

### P2 — Medium impact

#### [P2-1] Per-post Article schema for blog
- **Files:** `src/pages/blog/[slug].tsx` — pass `type="article"` + real `publishedTime`/`modifiedTime` to SEOHead.
- **Fix:** Hook SEOHead + StructuredData (type="Article") with real frontmatter from each blog post. Current fallback returns `wordCount: 1000` hardcoded — replace with real count or remove.

#### [P2-2] H1 / H2 audit
- **Rule:** One H1 per page, matching primary keyword. Module pages currently have H1 = module title (good) but verify H2/H3 contain secondary keywords naturally.
- **Grep:** `<h1` across `src/pages/**/*.tsx` — confirm exactly one per route.

#### [P2-3] `alt` attribute audit
- **Files:** `src/pages/modules/*/components/HeroSection.tsx` and similar — most use `alt={t("<module>.hero.imageAlt")}` (good). Verify those keys exist in all 3 locales with keyword-rich alt text (NOT just "Omniflow Logo" or "hero image").
- **Grep:** `<img` across `src/**/*.tsx`. Flag any alt="" or missing alt.

#### [P2-4] Bundle size (LCP impact)
- **Current:** `dist/assets/index-*.js` = **1.35 MB uncompressed**, 287 KB gzip. Above Vite's 500 KB chunk warning.
- **Fix:** Route-based code splitting via `React.lazy()` on module pages. 25 module routes × ~60 KB each ≈ 1.5 MB — all eagerly loaded today. Expected LCP improvement once split.
- **Watch-out:** measure with Lighthouse before/after; bundled-size drop doesn't always move LCP in practice.

#### [P2-5] Update `public/llms.txt`
- **File:** `public/llms.txt` — currently exists but content unknown to this plan. Should summarize product, modules list, and link to docs so LLM-based search tools (Perplexity, ChatGPT browse) can cite correctly.
- **Template:** title + 1-paragraph elevator pitch + bulleted list of 25 modules with one-line description each + contact URL.

### P3 — Low / nice-to-have

- **[P3-1]** Add `og:image:width` / `og:image:height` meta (required by Facebook's scraper for first-time shares).
- **[P3-2]** `application-name` meta + `apple-mobile-web-app-title` for PWA install UX.
- **[P3-3]** Per-locale `twitter:site` (currently all `@omniflowid`) if separate handles exist.
- **[P3-4]** Register site with Google Search Console, Bing Webmaster, IndexNow API.
- **[P3-5]** Blog content calendar targeting § 1.3 supporting keywords — 1 post/week, 1200+ words, internal-link to matching module page.
- **[P3-6]** Add `dateModified` driven by git commit time on each route (not just `new Date()` at render).

## 5. Validation checklist

After each change batch, run:

1. `bun run build` — must pass clean. Sitemap prebuild auto-runs.
2. `bunx lighthouse https://omniflow.id/id --preset=desktop --only-categories=seo,performance,accessibility` (after deploy).
3. Google Rich Results Test (https://search.google.com/test/rich-results) on:
   - `https://omniflow.id/id` — should detect WebPage, Organization, WebSite, SoftwareApplication.
   - `https://omniflow.id/id/modules/hris` — should detect WebPage, BreadcrumbList (after P1-2).
   - A blog post URL — should detect Article (after P2-1).
4. `view-source:https://omniflow.id/id` — confirm:
   - Exactly one `<title>` with the primary keyword.
   - Exactly one `<meta name="description">`.
   - Exactly one `<link rel="canonical">`.
   - 4 `<link rel="alternate" hreflang="...">` (en, id, zh, x-default).
   - No duplicate `Organization` JSON-LD (grep for `"@type": "Organization"` — should appear once max).
5. Fetch the 4 sitemaps and run through https://www.xml-sitemaps.com/validate-xml-sitemap.html.
6. Submit sitemaps to Google Search Console + Bing Webmaster Tools after deploy.

## 8. Hand-off notes

- All 25 modules + 4 categories are already in `src/lib/modules.ts`. Iterate over it to avoid drift.
- Sitemaps auto-regenerate on build; don't hand-edit the XML.
- i18n keys: if a module's `seo.title` key is missing in one locale, i18n will fall back to the key name (literal string). Always add all 3 locales together.
- Commit in small, reviewable batches grouped by P0/P1/P2. One big SEO PR is a merge-review nightmare.
- When editing `StructuredData.tsx`, beware of the `...data` spread at the end of each builder — caller-supplied `data` can override schema fields. Usually harmless but worth knowing.
